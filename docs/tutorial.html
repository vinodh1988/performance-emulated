1. Project Purpose

performance-all-round is a dashboard plus tutorial project. It is meant to be useful during real MongoDB administration practice, not only as a static document.

LayerWhat It Does
Web dashboardRuns MongoDB performance checks through the MongoDB driver and shows output in the browser.
Synthetic loadCreates realistic collections and indexes so slow query, profiler, stats, and monitoring checks have meaningful data.
Command scriptsReusable MongoDB scripts for load generation and slow query comparison.
TutorialStep-by-step commands, interpretation notes, and screenshot panels for every major topic.

2. Run the Dashboard Locally

The app uses the MongoDB Node.js driver. Install dependencies once, then connect directly to the replica set with URI, username/password, and TLS certificate paths.

cd /path/to/performance-all-round
npm install

PERF_MONGO_URI="mongodb://host1:27017,host2:27017,host3:27017/admin?replicaSet=rsTraining&authSource=admin&tls=true" \
PERF_MONGO_USER="siteAdmin" \
PERF_MONGO_PASSWORD="<admin-password>" \
PERF_TLS_CA_FILE="/path/to/mongodb-ca.crt" \
PERF_TLS_PEM_KEY_FILE="/path/to/windows-client.pem" \
npm start

Open the dashboard:

http://localhost:3010

Dashboard Buttons

  • Replica Health checks hello and replSetGetStatus.
  • Server Status captures connections, op counters, memory, WiredTiger cache, and locks.
  • MongoDB Logs uses the getLog admin command.
  • API Bottlenecks reports MongoDB memory, cache, lock, connection, and storage indicators.
  • Storage Stats uses dbStats and collStats.
  • Generate Load creates the lab database and indexes.
  • Run Profiler Slow Query compares before and after index performance.

3. Dockerized Run

The container connects directly to MongoDB. Mount a local certificate folder into /certs and provide the MongoDB URI and credentials through .env.

cd /path/to/performance-all-round
mkdir -p certs
cp /path/to/mongodb-ca.crt ./certs/mongodb-ca.crt
cp /path/to/windows-client.pem ./certs/windows-client.pem
chmod 444 ./certs/mongodb-ca.crt
chmod 400 ./certs/windows-client.pem
cp .env.example .env
nano .env

docker compose up --build -d

Example .env

PERF_MONGO_URI=mongodb://host1:27017,host2:27017,host3:27017/admin?replicaSet=rsTraining&authSource=admin&tls=true
PERF_MONGO_USER=siteAdmin
PERF_MONGO_PASSWORD=<admin-password>
PERF_AUTH_DB=admin
PERF_CERTS_DIR=./certs
PERF_TLS_CA_FILE=/certs/mongodb-ca.crt
PERF_TLS_PEM_KEY_FILE=/certs/windows-client.pem
PERF_LAB_DB=performance_all_round_lab

4. Synthetic Load Generation

The dashboard runs scripts/synthetic_load.js on the MongoDB primary. The script creates a clean lab database named performance_all_round_lab by default.

The synthetic load script drops and recreates only the configured lab database. Keep the default lab database name unless you intentionally want to recreate another database.
// Default collections
customers: 20000

// Main tuning index
db.orders.createIndex(
  { region: 1, status: 1, amount: 1, createdAt: -1 },

Why This Data Is Useful

  • orders supports slow query tuning, index comparison, and stats.
  • events supports high-volume monitoring and TTL examples.
  • customers supports partial index and lookup-style workloads.

5. Log Analysis Commands

Logs are the best source when you need to know exactly what MongoDB recorded: slow query, TLS failure, authentication, index build, restore, import, network disconnect, or storage warning.

db.adminCommand({ getLog: "global" })
db.adminCommand({ getLog: "startupWarnings" })

How to Read Log Evidence

  • planSummary: COLLSCAN points to a query that scanned a collection.
  • docsExamined shows how much data MongoDB had to inspect.
  • SSLHandshakeFailed usually means the client did not use TLS or used the wrong certificate chain.
  • Index build messages confirm index creation activity.

6. Performance Checking

Performance checking is a sequence: identify the expensive operation, read the plan, check server pressure, then fix the query or index.

db.serverStatus()
db.getSiblingDB("admin").aggregate([{ $currentOp: { allUsers: true, idleConnections: false } }])

db.orders.find({
  region: "HYD",
  status: "PAID",
  amount: { $gt: 100000 }
}).sort({ createdAt: -1 }).limit(25).explain("executionStats")
MetricMeaning
totalDocsExaminedDocuments MongoDB inspected. Lower is usually better.
totalKeysExaminedIndex keys MongoDB inspected.
executionTimeMillisTime spent executing the operation.
planSummaryQuick signal such as COLLSCAN or IXSCAN.

7. Memory Bottlenecks

Memory bottlenecks appear when MongoDB and the OS cannot keep the active working set in RAM. Watch WiredTiger cache, available memory, dirty bytes, eviction pressure, and swap.

const s = db.serverStatus()
printjson({
  mem: s.mem,
  cacheBytes: s.wiredTiger.cache["bytes currently in the cache"],
  maxCacheBytes: s.wiredTiger.cache["maximum bytes configured"],
  dirtyBytes: s.wiredTiger.cache["tracked dirty bytes in the cache"],
  pagesRead: s.wiredTiger.cache["pages read into cache"],
  pagesWritten: s.wiredTiger.cache["pages written from cache"]
})

If cache is near maximum and pages read from disk keep increasing quickly, the workload may be larger than memory or queries may be scanning too much data.

8. Storage Bottlenecks

Storage bottlenecks can come from low disk space, slow disk reads, heavy checkpoints, large indexes, or write bursts.

use performance_all_round_lab
db.stats(1024 * 1024)
db.orders.stats(1024 * 1024)
db.events.stats(1024 * 1024)

Signals

  • Root disk usage above a safe threshold.
  • Index size growing faster than expected.
  • Slow query logs with large bytesRead.
  • High disk wait in OS tools.

9. Profiling and Slow Query Performance

The profiler should be enabled for a short investigation window. The project script enables profiler, forces a baseline query, creates the supporting index, compares results, reads system.profile, and disables profiler.

use performance_all_round_lab
db.setProfilingLevel(1, { slowms: 10, sampleRate: 1.0 })

const filter = { region: "HYD", status: "PAID", amount: { $gt: 100000 } }
db.orders.find(filter).sort({ createdAt: -1 }).limit(25).explain("executionStats")

db.orders.createIndex(
  { region: 1, status: 1, amount: 1, createdAt: -1 },
  { name: "idx_region_status_amount_createdAt" }
)

db.orders.find(filter).sort({ createdAt: -1 }).limit(25).explain("executionStats")
db.system.profile.find({}, { ts: 1, millis: 1, docsExamined: 1, keysExamined: 1, planSummary: 1 }).sort({ ts: -1 }).limit(10)
db.setProfilingLevel(0)

10. Replica Set Health and Lag

Performance can look bad when secondaries lag, elections happen, or clients cannot reach advertised replica set members. Check replica health together with server metrics.

rs.status().members.map(m => ({
  name: m.name,
  stateStr: m.stateStr,
  health: m.health,
  optimeDate: m.optimeDate,
  pingMs: m.pingMs,
  syncSourceHost: m.syncSourceHost || ""
}))

rs.printReplicationInfo()
rs.printSecondaryReplicationInfo()

11. All-Round Troubleshooting Runbook

  1. Confirm the primary with db.hello().
  2. Check replica set health and lag with rs.status().
  3. Run dashboard Server Status to inspect connections, op counters, memory, cache, and locks.
  4. Run dashboard Analyze Logs to inspect slow query, TLS, auth, and index build messages.
  5. Run dashboard API Bottlenecks for MongoDB memory, cache, lock, and storage indicators.
  6. Use serverStatus counters for live server activity.
  7. Use profiler, current operations, and collection stats to identify hot namespaces.
  8. Enable profiler briefly and capture system.profile.
  9. Use explain("executionStats") to compare query plans.
  10. Create or adjust indexes, then re-test the same query.
  11. Disable profiler after collecting evidence.
